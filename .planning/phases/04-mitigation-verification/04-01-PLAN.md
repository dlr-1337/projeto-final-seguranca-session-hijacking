# Plan 04-01: Create Mitigation Verification Checklist

<objective>
As a demo presenter, I want a repeatable mitigation verification checklist and focused regression proof so that the team can show the same Session Hijacking story succeeding before mitigation and failing after the implemented session protections make the reused cookie invalid.

This plan covers `FIX-03` by creating the main Phase 4 checklist and test evidence. It intentionally verifies the mitigated scenarios that Phase 3 actually created: obsolete vulnerable cookies in fixed mode and fixed cookies after logout/invalidation.
</objective>

<threat_model>
## Threats

| ID | Threat | Severity | Mitigation in this plan |
|----|--------|----------|-------------------------|
| T-04-01 | The proof overclaims that `HttpOnly` blocks replay of an active manually copied cookie. | high | Docs and test names frame the denial around obsolete, logged-out, or expired sessions. |
| T-04-02 | The fixed-mode HTTP fallback is mistaken for complete secure transport. | high | Checklist separates `npm run dev:fixed` inspection fallback from secure-path test proof and states `Secure` requires HTTPS. |
| T-04-03 | New tests accidentally remove or weaken the vulnerable before proof. | high | Keep `tests/session-reuse-attack.test.js` passing and include vulnerable success as the before baseline. |
| T-04-04 | The checklist drifts into external targets or real data. | high | All steps use `127.0.0.1`, fake users, local clients, and ethical warnings. |

## Residual Risk

An active stolen session cookie is still a bearer credential until it expires or is invalidated. This residual risk is documented rather than hidden.
</threat_model>
